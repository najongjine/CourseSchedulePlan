먼저 감각으로 이해하기

쇼핑몰의 구매 흐름이 처음 끝까지 이어지는 날

화려한 결제 화면보다 중요한 것은 선택한 상품이 정확한 가격과 수량으로 주문표에 남는가이다.

오늘 시연은 화면과 SQL 결과를 함께 보여 주어 기능이 실제 저장으로 이어졌음을 확인한다.

목록상품 선택→cart총액→POST주문→SQL내역 확인

손가락 실습: 시연 전에 입력값, 예상 총액, 저장될 상세 행을 종이에 적고 실제 결과와 하나씩 비교한다.

직접 따라치기

주문 버튼 함수의 학생 이해 범위 핵심을 정리한다.

코드를 입력한 뒤 반드시 실행한다

async function submitOrder() {
  if (!customerName.trim() || cart.length === 0) {
    alert("주문자와 상품을 확인하세요.");
    return;
  }
  const order = {
    customer_name: customerName,
    items: cart.map(item => ({
      product_id: item.product_id,
      quantity: 1,
      unit_price: item.price
    }))
  };
  await fetch("http://localhost:8000/orders", {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify(order)
  });
  setCart([]);
  alert("주문이 저장되었습니다.");
}

실행 결과 예

장바구니에 상품 담기 -> 총액 표시 -> 이름 입력 -> 주문 저장됨 알림 -> 장바구니 비워짐

한 줄씩 읽어 보기

  1. 검증을 통과한 요청만 서버로 전송된다.
  2. 주문 완료 뒤 화면 cart를 비우지만, DB orders와 order_items 데이터는 남아 있다.
  3. 따라서 저장 확인은 화면이 아니라 SQL JOIN 결과로 한다.
입력한 뒤 바꾸어 볼 값

주문 완료 뒤 장바구니만 비어 있고 DB 주문은 남는 차이를 그림으로 표현한다.

손으로 추적하는 문제

화면 상태와 영구 저장 데이터의 차이는 데이터 입출력과 DB 이해의 핵심이다.

SELECT SUM(i.quantity * i.unit_price) AS total
FROM orders o
JOIN order_items i ON o.order_id = i.order_id
WHERE o.customer_name = '민지';

문제. 이 조회는 무엇을 계산하는가?

정답과 풀이 열기

민지의 주문상세 행 금액들을 합쳐 저장된 구매 총액을 계산한다.

감각을 굳히는 연습

먼저 종이에 예상 결과를 적고, 그 다음 코드를 바꾸어 실행해서 확인한다. 틀렸다면 틀린 이유가 학습 기록이다.

  1. 상품 두 개를 담은 예상 총액을 적고 화면 총액과 맞춘다.
  2. 주문 후 JOIN 조회와 SUM 조회 결과를 확인한다.
  3. 빈 이름, 빈 장바구니 테스트를 기록한다.
