완성: 주문과 매출 분석 DB

상품·주문·주문상세 관계를 이용해 주문 내역과 매출 질문에 SQL로 답하는 프로젝트를 완성한다.

먼저 감각으로 이해하기

DB는 화면 뒤에서 답을 만드는 엔진

아직 React 화면은 없지만, 상품 목록과 주문 내역과 매출을 모두 데이터로 다룰 수 있다.

다음 주 화면을 붙일 때도 DB 구조는 그대로이고, 조회 결과를 사용자에게 보기 좋게 보여 줄 뿐이다.

products상품→orders/items주문→JOIN내역→SUM매출

손가락 실습: 상품 조회 담당, 주문 입력 담당, JOIN 담당, 매출 담당으로 한 번씩 돌아가며 SQL을 실행한다.

직접 따라치기

핵심 시연용 SQL은 저장된 주문에 대한 사람 읽기 결과와 상품별 매출을 보여 준다.

코드를 입력한 뒤 반드시 실행한다

SELECT o.customer_name, p.name, i.quantity,
       i.quantity * i.unit_price AS subtotal
FROM orders o
JOIN order_items i ON o.order_id = i.order_id
JOIN products p ON p.product_id = i.product_id
ORDER BY o.order_id;

SELECT p.name, SUM(i.quantity) AS sold,
       SUM(i.quantity * i.unit_price) AS sales
FROM products p
JOIN order_items i ON p.product_id = i.product_id
GROUP BY p.product_id, p.name
ORDER BY sales DESC;

실행 결과 예

-- 첫 표: 주문자별 구매 품목과 줄 금액 -- 둘째 표: 상품별 총 판매량과 매출 내림차순

한 줄씩 읽어 보기

  1. 첫 SQL은 상세한 영수증 줄을 보여 준다.
  2. 둘째 SQL은 같은 상품 줄을 모아 분석 결과를 만든다.
  3. 두 결과 모두 order_items가 products에 올바른 외래키로 연결되어야 의미가 있다.
입력한 뒤 바꾸어 볼 값

선택 기능으로 재고 감소 UPDATE를 작성하되, 주문 저장과 재고 감소가 함께 성공해야 한다는 트랜잭션 개념을 설명만 한다.

손으로 추적하는 문제

관계형 SQL 문제에서는 표를 작게 그린 뒤 JOIN 결과 행을 만들고 마지막에 집계한다.

-- A 상품의 상세 행: 수량 2 단가 1000, 수량 3 단가 1000
SELECT SUM(quantity * unit_price)
FROM order_items
WHERE product_id = 1;

문제. 결과 매출은 얼마인가?

정답과 풀이 열기

두 행의 금액은 2000과 3000이며 SUM 결과는 `5000`이다.

감각을 굳히는 연습

먼저 종이에 예상 결과를 적고, 그 다음 코드를 바꾸어 실행해서 확인한다. 틀렸다면 틀린 이유가 학습 기록이다.

  1. 주문 한 건을 추가하고 JOIN 조회 결과가 늘어나는지 확인한다.
  2. 상품별 판매량/매출 SQL을 시연한다.
  3. 상품-주문-상세 세 표를 분리한 이유를 그림으로 발표한다.
연습 확인 포인트

React 구간에서는 전체 SQL을 다시 쓰지 않고 이 데이터가 화면으로 이동하는 경로를 이해한다.
