테이블은 상품을 쌓는 표

코드 안 딕셔너리에 있던 상품 데이터를 PostgreSQL의 `products` 테이블로 옮길 설계부터 시작한다.

7주차 1일차오늘의 목표4시간 흐름감각으로 이해하기직접 따라치기기사형 코드 추적손으로 푸는 연습금요일 프로젝트 연결
이전다음

오늘의 목표

  • 행과 열을 구분한다.
  • 기본키의 필요성을 이해한다.
  • 표준 SQL 중심의 CREATE TABLE을 작성한다.

오늘 4시간의 흐름

1교시개념을 그림과 코드로 이해
2교시예제를 직접 입력하고 실행
3교시금요일 작품 기능 한 조각 작성
4교시값을 바꾸고 손 추적 연습

먼저 감각으로 이해하기

테이블은 같은 종류의 카드가 줄 맞춰 들어가는 서랍장

한 행은 상품 하나이고, 열은 모든 상품이 공통으로 가져야 하는 항목이다.

이름이 같아질 수도 있으므로 상품을 정확히 가리킬 고유 번호인 기본키를 둔다.

행상품 한 개→열name/price/stock→기본키product_id→DB저장

손가락 실습: 상품 카드 세 장을 그리고, 각 카드에 같은 열 제목을 적은 뒤 고유 번호를 붙인다.

직접 따라치기

PostgreSQL에서 새 데이터베이스에 접속한 뒤 테이블 하나를 생성한다.

코드를 입력한 뒤 반드시 실행한다

CREATE TABLE products (
    product_id INTEGER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    name VARCHAR(100) NOT NULL,
    price INTEGER NOT NULL,
    stock INTEGER NOT NULL
);

실행 결과 예

CREATE TABLE -- products 테이블이 생성됨

한 줄씩 읽어 보기

  1. `products`는 앞으로 상품을 보관할 테이블 이름이다.
  2. `GENERATED BY DEFAULT AS IDENTITY`는 입력하지 않아도 번호를 만들어 주는 표준 방식에 가까운 정의다.
  3. `NOT NULL`은 반드시 값이 있어야 하는 열이라는 규칙이다.
입력한 뒤 바꾸어 볼 값

price와 stock에 음수가 들어가면 안 된다는 생각을 적고, 나중 `CHECK` 조건으로 확장할 수 있음을 표시한다.

손으로 추적하는 문제

기사 SQL에서는 테이블 정의와 기본키/외래키의 의미를 이해해야 한다.

CREATE TABLE category (
    category_id INTEGER PRIMARY KEY,
    name VARCHAR(30) NOT NULL
);

문제. `category_id`와 `name`의 역할을 각각 설명하라.

정답과 풀이 열기

`category_id`는 각 행을 유일하게 식별하는 기본키이며, `name`은 비어 있을 수 없는 최대 30자 이름 열이다.

감각을 굳히는 연습

먼저 종이에 예상 결과를 적고, 그 다음 코드를 바꾸어 실행해서 확인한다. 틀렸다면 틀린 이유가 학습 기록이다.

  1. 종이에 products 행 3개를 그려 열 값을 채운다.
  2. 상품명, 가격, 재고 중 비어 있으면 안 되는 값에 이유를 단다.
  3. `SERIAL`이 아닌 IDENTITY를 쓰는 이유를 표준 SQL 학습 원칙과 연결해 적는다.
