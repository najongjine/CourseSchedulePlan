먼저 감각으로 이해하기

한 주문 저장은 머리 한 줄 다음 상세 여러 줄

서버는 먼저 orders에 주문자를 저장하여 새 주문 번호를 얻고, cart 상품마다 같은 번호로 order_items를 저장한다.

서버 뼈대와 트랜잭션 처리는 강사가 제공한다. 학생은 두 INSERT의 역할과 화면의 POST 호출을 입력한다.

POST 주문JSON→orders새 ID→order_items여러 INSERT→JOIN저장 확인

손가락 실습: 빈 영수증 상단에 주문번호를 찍은 뒤 상세 상품 줄마다 같은 번호를 복사해 쓰는 모습을 재현한다.

실행 결과 예

주문하기 클릭 -> 주문 완료 문구 표시 SQL JOIN 조회 -> 주문자와 담은 상품 행 확인

한 줄씩 읽어 보기

  1. 첫 INSERT가 주문 한 건의 ID를 만든다.
  2. 둘째 INSERT는 cart 항목 수만큼 실행되어 같은 order_id에 연결된다.
  3. 실제 서버 코드에서는 드라이버에 맞는 매개변수 표시와 트랜잭션을 교사 뼈대가 처리한다.
입력한 뒤 바꾸어 볼 값

주문 실패 시 일부 상세만 저장되면 왜 문제인지 말하고, 트랜잭션이라는 용어를 메모한다.

손으로 추적하는 문제

주문 저장은 관계형 DB와 통합 구현을 한 장면에 보여 주며, 값 매개변수 사용도 유지한다.

SELECT o.customer_name, p.name, i.quantity
FROM orders o
JOIN order_items i ON o.order_id = i.order_id
JOIN products p ON p.product_id = i.product_id
WHERE o.customer_name = '민지';

문제. 이 SQL은 무엇을 확인하기 위한 문장인가?

감각을 굳히는 연습

먼저 종이에 예상 결과를 적고, 그 다음 코드를 바꾸어 실행해서 확인한다. 틀렸다면 틀린 이유가 학습 기록이다.

  1. 상품 두 개를 담아 주문 저장을 실행한다.
  2. JOIN SQL로 주문자가 선택한 상품이 저장됐는지 확인한다.
  3. 주문 저장 중 실패 시 되돌려야 하는 이유를 한 문장으로 쓴다.
