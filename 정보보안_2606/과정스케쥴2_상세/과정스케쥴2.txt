정보보안_2606 과정 상세 스케쥴 2안
정보처리기사 준비 + React 쇼핑몰 미니 프로젝트 과정


1. 과정 개요

- 대상: 코딩 경험이 거의 없는 비전공자 학생 4명
- 기간: 총 12주, 월요일~금요일, 1일 4시간
- 언어 학습 순서: C언어 -> Java -> Python -> SQL -> React 통합 프로젝트
- 데이터베이스: PostgreSQL
- 과정 목표:
  1) 정보처리기사 실기 프로그래밍 영역의 C, Java, Python, SQL 코드를 손으로 추적하고 제한 시간 안에 직접 작성할 수 있다.
  2) 월~목 기사형 문제 훈련에서 끝나지 않고, 금요일마다 요구사항만 보고 실행 가능한 미니 프로젝트를 처음부터 완성한다.
  3) 마지막 4주 동안 React, Python API, PostgreSQL을 연결한 작은 쇼핑몰을 완성한다.


2. 난이도와 운영 원칙

- 월요일부터 목요일까지는 `결과 예측 -> 오류 수정 -> 기능 직접 작성`의 세 단계를 반복하고, 금요일에는 완성 코드를 제공하지 않은 채 과제 요구사항만 제시한다.
- 1~6주차는 정보처리기사 코딩 숙련 구간으로 운영한다.
  - 매일 기사형 코드 추적 문제 5개 이상을 손으로 푼다.
  - 매일 1개 이상의 오류 수정 또는 빈칸 구현 문제를 컴파일/실행하여 검증한다.
  - 목요일에는 금요일 과제에 필요한 기능을 서로 다른 데이터로 다시 작성한다. 금요일 제출 코드를 미리 제공하지 않는다.
- 1~8주차 금요일 프로젝트는 학생이 처음부터 직접 입력할 수 있는 규모로 제한한다.
  - 첫 언어 주차: 완성 코드 약 70~120줄
  - 두 번째 언어 주차: 완성 코드 약 110~180줄
  - SQL 주차: 테이블 2~4개, 핵심 SQL 10~15문장
- 9~12주차 쇼핑몰은 전체 코드를 매번 새로 입력하지 않는다.
  - 교사가 실행 가능한 공통 프로젝트 뼈대를 제공한다.
  - 학생은 그 주에 배우는 컴포넌트, API 함수, SQL 문장만 직접 입력하고 실행한다.
  - 매주 금요일에는 그 주 기능이 실제 화면에서 동작하는 버전을 완성한다.
- 쇼핑몰은 교육용 최소 기능만 구현한다. 결제, 회원가입, 로그인, 이미지 업로드, 배포는 필수 범위에서 제외한다.
- 학생이 이해하지 못한 긴 코드를 복사하여 완성하는 방식은 사용하지 않는다.


3. PostgreSQL 및 SQL 작성 원칙

- SQL 문법은 가능한 한 표준 SQL 중심으로 수업한다.
- 기본 명령은 `CREATE TABLE`, `INSERT`, `SELECT`, `UPDATE`, `DELETE`, `JOIN`, `GROUP BY`, `ORDER BY`, `HAVING` 중심으로 구성한다.
- PostgreSQL에서 자동 증가 기본키가 필요할 때는 `SERIAL`보다 아래 형식을 우선 사용한다.

  `product_id INTEGER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY`

- `GENERATED ... AS IDENTITY`는 표준 SQL 방식에 가깝고 PostgreSQL에서도 지원하므로, 교육용 스키마에 적합하다.
- `SERIAL`, `RETURNING`, `ILIKE` 등 PostgreSQL 전용 기능을 소개할 경우에는 "PostgreSQL 확장 기능"이라고 구분하여 설명한다.

1주차: C 제어문과 연산 추적 - 점수 판정 및 통계 프로그램
============================================================

주차 목표
- C의 대입, 산술/관계/논리 연산, 증감 연산과 제어문의 실행 순서를 손으로 정확히 추적한다.
- 반복 입력과 누적 계산을 사용해 요구사항만 보고 작은 콘솔 프로그램을 직접 작성한다.

월요일 - 변수, 자료형, 연산 순서와 출력 추적
- 이론: `int`, `double`, 정수 나눗셈, 나머지, 형변환, 연산자 우선순위, 전위/후위 증감
- 기사형 추적: 대입과 증감 연산이 섞인 출력 결과 문제 10개
- 직접 작성: 국어/영어 점수를 입력받아 합계와 실수 평균 출력, 정수 나눗셈 오류 고치기
- 숙련 체크: 실행하지 않고 변수 변화표를 먼저 작성한 뒤 결과와 비교한다.

화요일 - 조건문과 논리식
- 이론: `if / else if / else`, 중첩 조건, `&&`, `||`, `!`, 조건 경계값
- 기사형 추적: 합격/불합격 및 장학 조건 논리식 문제 8개
- 직접 작성: 점수 범위 검사, 평균 등급, 과락 포함 합격 판정 함수 없는 버전 구현
- 오류 수정: `>=`와 `>`, `&&`와 `||`를 잘못 쓴 코드를 찾아 테스트한다.

수요일 - 반복문과 누적 계산
- 이론: `for`, `while`, `do while`, 반복 횟수, 누적합, 카운트, 최댓값/최솟값 갱신
- 기사형 추적: 반복 후 변수값, 중첩 반복 출력 형태, `break`/`continue` 문제 10개
- 직접 작성: 학생 5명의 점수를 순차 입력받아 합계, 평균, 합격자 수, 최고점 계산
- 프로젝트 준비: 배열 없이 한 명씩 입력받아도 통계를 만들 수 있음을 확인한다.

목요일 - 메뉴 반복과 제한 시간 구현
- 이론: 입력 검증, 센티널 값, 반복 메뉴, 처리 순서 분해
- 기사형 추적: 조건문과 반복문이 결합된 15~25줄 C 코드 5개
- 직접 작성: `1. 점수 입력  2. 결과 보기  0. 종료` 흐름을 40분 안에 다시 작성
- 과제 준비: 금요일 프로젝트 요구사항을 읽고 입력/처리/출력 테스트 항목만 설계한다. 완성 코드는 제공하지 않는다.

금요일 미니 프로젝트 과제 - "점수 판정 및 통계 프로그램"
- 과제 제시: 학생 수와 점수를 입력받아 성적 결과를 출력하는 프로그램을 처음부터 작성한다.
- 필수 기능: 학생 수 입력, 각 점수 범위 검증, 등급 판정, 합계/평균, 합격자 수, 최고점/최저점, 종료 출력
- 필수 문법: `if`, 반복문, 논리 연산, 누적 변수, 최댓값/최솟값 갱신
- 선택 기능: 재입력 처리 또는 평균에 따른 반 전체 평가 메시지
- 코드 규모 목표: 70~100줄
- 검증 데이터: 정상 점수, `0`/`100` 경계값, 범위 밖 입력, 전원 불합격 입력을 직접 시험한다.
- 학생 설명 포인트: 최고점이 바뀌는 조건, 평균 계산 시 형변환, 반복 종료 조건

목요일 - 구조체와 기능 분리

금요일 미니 프로젝트 과제 - "학생 성적 관리 프로그램"
- 과제 제시: 이름과 3과목 점수를 저장하고 조회/통계를 수행하는 관리 프로그램을 처음부터 작성한다.

6. 쇼핑몰 데이터베이스 최소 구성

필수 테이블은 3개만 사용한다.

1) products
- product_id: 상품 번호, 기본키, 자동 생성
- name: 상품명
- price: 가격
- stock: 재고

2) orders
- order_id: 주문 번호, 기본키, 자동 생성
- customer_name: 주문자 이름
- ordered_at: 주문 날짜/시간

3) order_items
- order_item_id: 주문상세 번호, 기본키, 자동 생성
- order_id: 어느 주문인지 나타내는 외래키
- product_id: 어느 상품인지 나타내는 외래키
- quantity: 주문 수량
- unit_price: 주문 당시 단가

최소 스키마 예시

CREATE TABLE products (
    product_id INTEGER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    name VARCHAR(100) NOT NULL,
    price INTEGER NOT NULL,
    stock INTEGER NOT NULL
);

CREATE TABLE orders (
    order_id INTEGER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    customer_name VARCHAR(50) NOT NULL,
    ordered_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE order_items (
    order_item_id INTEGER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    order_id INTEGER NOT NULL REFERENCES orders(order_id),
    product_id INTEGER NOT NULL REFERENCES products(product_id),
    quantity INTEGER NOT NULL,
    unit_price INTEGER NOT NULL
);
