# AI 시대에도 프로그래밍을 배워야 하는 이유

국비학원 비전공자 수강생을 위한 입문 교재

---

## 이 교재를 읽기 전에

요즘 개발을 시작하려는 사람들은 이런 말을 자주 듣는다.

```text
AI가 코드를 다 짜준다.
이제 개발자는 필요 없다.
비전공자는 더 늦었다.
개발자 취업은 끝났다.
```

이런 말을 들으면 불안해지는 것이 당연하다. 특히 비전공자로 국비학원 과정을 시작한 사람이라면 더 그렇다. 아직 변수, 조건문, 반복문도 익숙하지 않은데 AI가 이미 코드를 만든다고 하니 “내가 지금 배우는 게 의미가 있나?”라는 생각이 들 수 있다.

이 교재의 핵심 관점은 다음과 같다.

```text
개발자가 사라진 것이 아니라,
개발자의 역할이 바뀌고 있다.
```

AI는 코드를 생성할 수 있다. 하지만 무엇을 만들어야 하는지 정리하고, 그 코드가 맞는지 판단하고, 오류를 고치고, 보안을 확인하고, 데이터베이스와 API와 화면을 연결하고, 실제 서비스로 완성하는 일은 여전히 사람이 이해해야 한다.

프로그래밍을 배운다는 것은 단순히 문법을 외우는 일이 아니다. AI가 만든 결과를 읽고, 판단하고, 수정하고, 책임질 수 있는 사람이 되는 과정이다.

# 1장. “AI 때문에 개발자 망했다”는 말은 왜 나왔을까?

## 1.1 AI가 실제로 잘하는 것

AI는 이전보다 훨씬 자연스럽게 코드를 작성한다. 간단한 함수, 예제 코드, 반복적인 코드, 문서 요약, 오류 메시지 설명, 테스트 코드 초안 작성 같은 일에서 큰 도움을 준다.

예를 들어 다음과 같은 요청은 AI가 꽤 잘 처리한다.

```text
Python으로 숫자 리스트의 평균을 구하는 함수를 만들어줘.
JavaScript로 버튼을 클릭하면 문구가 바뀌게 해줘.
SQL에서 users 테이블의 active 사용자를 조회하는 쿼리를 작성해줘.
이 에러 메시지가 무슨 뜻인지 설명해줘.
```

이런 장면만 보면 “이제 사람이 코드를 배울 필요가 없나?”라는 생각이 들 수 있다.

하지만 실제 개발은 예제 코드 하나를 만드는 것보다 훨씬 복잡하다.

## 1.2 개발은 코드 작성만이 아니다

많은 사람이 개발을 이렇게 생각한다.

```text
개발 = 코드를 입력하는 일
```

하지만 실제 개발은 다음에 더 가깝다.

```text
개발 = 문제를 이해하고, 해결 방법을 설계하고,
코드로 구현하고, 검증하고, 운영 가능한 형태로 완성하는 일
```

코드 작성은 개발의 중요한 부분이지만 전부는 아니다.

개발자는 다음과 같은 질문에 답해야 한다.

- 사용자가 정말 원하는 것은 무엇인가?
- 어떤 기능을 먼저 만들어야 하는가?
- 데이터는 어디에 저장할 것인가?
- 화면과 서버는 어떻게 연결할 것인가?
- 로그인한 사용자만 접근하게 하려면 어떻게 해야 하는가?
- 오류가 났을 때 어떤 메시지를 보여줄 것인가?
- 이 코드가 안전한가?
- 이 기능이 나중에 수정하기 쉬운 구조인가?
- 배포 후 문제가 생기면 어디를 확인해야 하는가?

AI가 코드를 생성할 수 있어도, 이런 판단을 할 사람은 필요하다.

## 1.3 “개발자가 망했다”보다 정확한 표현

더 정확한 표현은 다음과 같다.

```text
단순 반복 코딩만 하는 방식은 점점 가치가 낮아지고 있다.
하지만 문제를 이해하고 AI를 활용해 결과물을 완성할 수 있는 개발자의 가치는 여전히 중요하다.
```

AI 때문에 개발자가 완전히 사라지는 것이 아니라, 개발자에게 요구되는 능력이 바뀌고 있다.

예전에는 코드를 얼마나 빨리 직접 치는지가 중요했다면, 이제는 다음 능력이 더 중요해지고 있다.

- 문제를 정확히 설명하는 능력
- AI가 만든 코드를 읽는 능력
- 틀린 부분을 찾는 능력
- 전체 구조를 이해하는 능력
- 보안과 데이터 흐름을 판단하는 능력
- 실제 서비스로 연결하는 능력

즉 AI 시대의 개발자는 “혼자 모든 코드를 손으로 치는 사람”이 아니라, “AI를 도구로 사용해 문제를 해결하는 사람”에 가까워지고 있다.

# 2장. AI가 코드를 짜주는데 왜 프로그래밍을 배워야 할까?

## 2.1 AI의 답이 항상 맞지는 않다

AI는 그럴듯한 답을 만들 수 있다. 문제는 그 답이 항상 맞지는 않다는 것이다.

AI가 만든 코드에는 다음 문제가 있을 수 있다.

- 실행이 안 된다.
- 일부 상황에서만 맞고 다른 상황에서는 틀린다.
- 보안상 위험하다.
- 너무 오래된 방식이다.
- 프로젝트 구조와 맞지 않는다.
- 데이터베이스 구조를 잘못 이해했다.
- 에러 처리가 빠져 있다.
- 성능이 나쁘다.
- 사용자의 요구사항과 다르다.

초보자에게 가장 위험한 상황은 “틀린 코드를 맞는 코드처럼 믿는 것”이다.

AI를 잘 쓰려면 AI의 답을 검토할 수 있어야 한다. 검토하려면 기본기를 배워야 한다.

## 2.2 계산기를 쓰려면 사칙연산을 알아야 한다

계산기가 있다고 해서 사칙연산을 전혀 몰라도 되는 것은 아니다.

계산기는 빠르게 계산해주지만, 어떤 계산을 해야 하는지 결정하는 것은 사람이다.

예를 들어 물건 가격이 30,000원이고 20% 할인이라면 계산기에 무엇을 입력해야 할까?

```text
30000 * 0.8
```

이 생각을 하려면 퍼센트와 곱셈의 의미를 알아야 한다.

AI도 비슷하다.

AI가 코드를 빠르게 만들어줄 수는 있지만, 무엇을 요청해야 하는지, 결과가 맞는지, 어디를 고쳐야 하는지는 사람이 알아야 한다.

## 2.3 내비게이션을 써도 운전은 알아야 한다

내비게이션은 길을 안내해준다. 하지만 운전을 대신 완전히 해주지는 않는다.

운전자는 여전히 다음을 판단해야 한다.

- 신호를 지켜야 한다.
- 앞차와 거리를 유지해야 한다.
- 도로 상황을 봐야 한다.
- 사고 위험을 피해야 한다.
- 내비게이션이 이상한 길을 안내하면 판단해야 한다.

AI는 개발자의 내비게이션과 비슷하다.

AI가 방향을 제안하고 코드를 만들어줄 수는 있지만, 개발자는 전체 상황을 보고 판단해야 한다.

프로그래밍을 배운다는 것은 AI가 안내하는 길을 이해하고, 위험한 길을 피하고, 목적지까지 도착할 수 있는 능력을 기르는 것이다.

## 2.4 코드를 읽지 못하면 수정할 수 없다

AI가 만들어준 코드를 그대로 붙여 넣었는데 에러가 났다고 하자.

이때 코드를 읽을 수 없다면 할 수 있는 일이 거의 없다.

```text
AI에게 다시 물어본다.
또 다른 코드를 받는다.
다시 붙여 넣는다.
다시 에러가 난다.
어디가 문제인지 모른다.
```

이 반복에 빠지면 AI를 쓰고 있어도 개발 실력은 늘지 않는다.

반대로 기본 문법과 구조를 이해하면 AI를 훨씬 잘 활용할 수 있다.

```text
이 함수에서 매개변수 이름이 잘못됐구나.
여기서 null 처리가 빠졌구나.
API 주소가 우리 프로젝트와 다르구나.
데이터베이스 컬럼명이 실제 테이블과 다르구나.
```

AI 시대에는 코드를 외워서 쓰는 능력보다 코드를 읽고 판단하는 능력이 더 중요해진다.

# 3장. AI가 해주기 어려운 개발자의 일

## 3.1 문제 정의

개발의 시작은 코딩이 아니라 문제 정의다.

예를 들어 누군가 이렇게 말할 수 있다.

```text
우리 학원 수강생 관리 프로그램을 만들어주세요.
```

이 말만으로는 개발을 시작하기 어렵다.

개발자는 질문해야 한다.

- 수강생의 어떤 정보를 관리해야 하는가?
- 출석도 관리하는가?
- 상담 이력도 필요한가?
- 강사는 어떤 화면을 보는가?
- 관리자는 어떤 권한을 가지는가?
- 엑셀 다운로드가 필요한가?
- 모바일에서도 사용해야 하는가?
- 개인정보는 어떻게 보호할 것인가?

AI에게 “수강생 관리 프로그램 만들어줘”라고 입력하면 코드가 나올 수는 있다. 하지만 실제로 필요한 프로그램인지, 현장에서 쓸 수 있는지는 별개의 문제다.

문제를 제대로 정의하지 못하면 AI가 아무리 빠르게 코드를 만들어도 엉뚱한 결과물이 나온다.

## 3.2 요구사항 분석

요구사항은 “프로그램이 해야 하는 일”이다.

예를 들어 게시판의 요구사항은 다음처럼 정리할 수 있다.

```text
사용자는 게시글 목록을 볼 수 있다.
로그인한 사용자는 게시글을 작성할 수 있다.
작성자는 자신의 게시글을 수정할 수 있다.
관리자는 모든 게시글을 삭제할 수 있다.
게시글 제목은 비어 있으면 안 된다.
```

여기서 중요한 점은 “누가”, “무엇을”, “어떤 조건에서” 할 수 있는지를 분명히 하는 것이다.

AI는 요구사항이 명확할수록 좋은 결과를 만든다. 반대로 요구사항이 흐릿하면 결과도 흐릿하다.

개발자는 AI에게 명령하는 사람이기 전에, 요구사항을 정리하는 사람이어야 한다.

## 3.3 코드 검토

AI가 만든 코드는 검토가 필요하다.

코드 검토에서는 다음을 확인한다.

- 요구사항을 만족하는가?
- 실행했을 때 오류가 없는가?
- 예외 상황을 처리하는가?
- 읽기 쉬운가?
- 중복이 너무 많지 않은가?
- 보안 문제가 없는가?
- 프로젝트의 기존 코드 스타일과 맞는가?

예를 들어 AI가 로그인 코드를 만들었다고 하자.

확인해야 할 것은 단순히 “로그인이 되는가?”만이 아니다.

```text
비밀번호를 평문으로 저장하고 있지 않은가?
없는 사용자가 로그인할 때 오류가 안전하게 처리되는가?
로그인 실패 횟수를 제한할 필요는 없는가?
인증된 사용자 정보가 안전하게 유지되는가?
```

코드가 돌아간다는 것과 안전하게 동작한다는 것은 다르다.

## 3.4 오류 수정

실제 개발에서는 에러가 자주 발생한다.

에러의 원인은 다양하다.

- 오타
- 타입 불일치
- 잘못된 경로
- 데이터베이스 연결 실패
- API 주소 오류
- 권한 문제
- 라이브러리 버전 차이
- 환경 변수 누락
- 서버 설정 문제

AI에게 에러 메시지를 물어볼 수는 있다. 하지만 에러가 발생한 프로젝트의 구조, 실행 환경, 데이터 흐름을 이해하지 못하면 해결이 어렵다.

개발자는 에러 메시지를 읽고, 원인을 좁히고, 수정한 뒤 다시 검증할 수 있어야 한다.

## 3.5 보안 판단

보안은 AI 시대에 더 중요해진다.

AI가 만든 코드가 편리해 보여도 다음 문제가 있을 수 있다.

- 비밀번호를 그대로 저장한다.
- 관리자 권한 검사를 빠뜨린다.
- 사용자의 입력값을 검증하지 않는다.
- 중요한 API 키를 코드에 직접 적는다.
- 누구나 다른 사람의 데이터를 조회할 수 있게 만든다.

예를 들어 다음과 같은 코드는 위험하다.

```text
사용자가 보낸 userId 값만 믿고 해당 사용자의 정보를 수정한다.
```

사용자가 자기 ID가 아니라 다른 사람의 ID를 보내면 어떻게 될까? 권한 검사가 없으면 다른 사람의 정보를 바꿀 수도 있다.

AI가 코드를 만들 수 있어도, 보안상 괜찮은지 판단하는 능력은 개발자에게 필요하다.

## 3.6 연결과 통합

현대 서비스는 여러 부분이 연결되어 있다.

```text
프론트엔드 화면
   ↓
API 요청
   ↓
백엔드 서버
   ↓
데이터베이스
   ↓
다시 백엔드
   ↓
다시 프론트엔드 화면
```

AI가 각각의 조각 코드를 만들어줄 수는 있다. 하지만 조각을 연결해 하나의 서비스로 만드는 일은 여전히 어렵다.

예를 들어 회원가입 기능 하나에도 여러 요소가 필요하다.

- 입력 화면
- 유효성 검사
- API 요청
- 서버 라우팅
- 비밀번호 암호화
- 데이터베이스 저장
- 중복 이메일 검사
- 성공/실패 응답
- 화면 메시지 처리

개발자는 이 흐름을 이해해야 한다.

# 4장. AI 시대에 프로그래밍 기본기가 더 중요한 이유

## 4.1 변수, 조건문, 반복문은 여전히 기본이다

AI가 등장해도 프로그램의 기본 구조는 크게 달라지지 않았다.

프로그램은 여전히 데이터를 저장하고, 조건을 판단하고, 반복 작업을 수행한다.

```python
score = 85

if score >= 60:
    print("합격")
else:
    print("불합격")
```

이 코드는 단순하지만 중요한 원리를 담고 있다.

```text
값을 저장한다.
조건을 판단한다.
상황에 따라 다른 결과를 낸다.
```

AI가 더 복잡한 코드를 만들어도 그 안에는 결국 변수, 조건문, 반복문, 함수, 객체, 자료구조가 들어 있다.

기초를 모르면 AI가 만든 코드도 읽을 수 없다.

## 4.2 함수와 객체는 큰 프로그램을 이해하는 도구다

작은 코드는 한눈에 볼 수 있다. 하지만 실제 프로젝트는 수십 개, 수백 개의 파일로 구성된다.

이때 함수와 객체를 이해해야 코드를 따라갈 수 있다.

```java
public class UserService {
    public void register(User user) {
        // 회원가입 처리
    }
}
```

이 코드를 보며 다음을 이해할 수 있어야 한다.

- UserService는 회원 관련 기능을 담당한다.
- register는 회원가입 기능이다.
- User는 회원 데이터를 담는다.

AI가 만든 프로젝트를 검토할 때도 이런 구조를 읽는 능력이 필요하다.

## 4.3 데이터베이스를 모르면 서비스가 완성되지 않는다

대부분의 서비스는 데이터를 저장한다.

- 회원 정보
- 게시글
- 댓글
- 상품
- 주문
- 결제
- 출석
- 성적
- 상담 기록

프로그래밍 언어만 알아서는 부족하다. 데이터베이스와 SQL을 이해해야 한다.

```sql
SELECT *
FROM students
WHERE course_id = 3;
```

AI에게 SQL을 만들어달라고 할 수는 있다. 하지만 테이블 구조를 이해하지 못하면 그 SQL이 맞는지 판단할 수 없다.

데이터베이스는 실제 서비스의 중심이다. AI 시대에도 데이터 구조를 이해하는 개발자는 필요하다.

## 4.4 API를 모르면 화면과 서버를 연결할 수 없다

웹 서비스에서 프론트엔드와 백엔드는 API로 대화한다.

```text
프론트엔드:
회원가입 정보를 서버로 보낸다.

백엔드:
정보를 검증하고 데이터베이스에 저장한다.

프론트엔드:
서버 응답에 따라 성공 또는 실패 메시지를 보여준다.
```

API를 이해하지 못하면 화면은 만들었지만 데이터가 저장되지 않거나, 서버는 만들었지만 화면과 연결되지 않는 문제가 생긴다.

AI가 각 부분의 코드를 만들어도, API 요청과 응답의 흐름을 이해해야 실제 기능을 완성할 수 있다.

## 4.5 프론트엔드와 백엔드는 함께 이해해야 한다

초보자는 프론트엔드와 백엔드를 따로 생각하기 쉽다.

하지만 실제 기능은 둘이 연결되어야 완성된다.

예를 들어 로그인 기능은 다음 요소가 함께 동작해야 한다.

```text
프론트엔드:
아이디와 비밀번호 입력
로그인 버튼 클릭
서버에 요청 전송

백엔드:
사용자 조회
비밀번호 확인
인증 처리
응답 반환

프론트엔드:
성공하면 메인 화면 이동
실패하면 오류 메시지 표시
```

AI 시대의 개발자는 “한 조각 코드”보다 “전체 흐름”을 이해해야 한다.

# 5장. AI를 잘 쓰는 사람과 못 쓰는 사람의 차이

## 5.1 질문의 수준이 결과의 수준을 결정한다

AI는 질문이 구체적일수록 더 좋은 답을 준다.

나쁜 질문의 예:

```text
게시판 만들어줘.
```

조금 나은 질문:

```text
Spring Boot와 MySQL을 사용해서 게시글 목록 조회 API를 만들어줘.
게시글은 id, title, content, author, created_at 필드를 가진다.
최신순으로 정렬하고, 한 페이지에 10개씩 조회하게 해줘.
```

두 질문의 차이는 요구사항의 구체성이다.

프로그래밍을 배운 사람은 더 정확히 질문할 수 있다.

## 5.2 AI 답변을 그대로 믿지 않는다

AI를 잘 쓰는 사람은 답을 받자마자 붙여 넣지 않는다.

먼저 확인한다.

```text
우리 프로젝트 언어와 맞는가?
라이브러리 버전과 맞는가?
파일 위치가 맞는가?
보안상 문제는 없는가?
예외 처리가 있는가?
테스트할 수 있는가?
```

AI 답변은 최종 답안이 아니라 초안에 가깝다.

개발자는 AI가 만든 초안을 검토하고 다듬어 실제 코드로 만든다.

## 5.3 작은 단위로 요청한다

초보자는 AI에게 한 번에 너무 큰 것을 요청하기 쉽다.

```text
쇼핑몰 전체 만들어줘.
```

이렇게 요청하면 결과가 크고 복잡해서 검토하기 어렵다.

더 좋은 방식은 기능을 작게 나누는 것이다.

```text
1. 상품 목록 데이터 구조를 설계한다.
2. 상품 목록 조회 API를 만든다.
3. 상품 목록 화면을 만든다.
4. 화면에서 API를 호출한다.
5. 로딩과 오류 상태를 처리한다.
```

프로그래밍 기본기를 배우면 문제를 작은 단위로 나누는 능력이 생긴다.

## 5.4 에러 메시지를 함께 제공한다

AI에게 질문할 때는 상황을 자세히 알려줘야 한다.

좋지 않은 질문:

```text
안 돼요. 고쳐주세요.
```

좋은 질문:

```text
Spring Boot에서 회원가입 API를 실행했는데 500 에러가 발생합니다.
콘솔에는 Duplicate entry for key email 이라고 나옵니다.
users 테이블의 email 컬럼에는 unique 제약조건이 있습니다.
중복 이메일일 때 500이 아니라 400 응답을 보내고 싶습니다.
```

이렇게 질문하려면 에러 메시지, 데이터베이스, HTTP 상태 코드, 제약조건을 어느 정도 이해해야 한다.

AI를 잘 쓰려면 결국 개발 기초가 필요하다.

# 6장. 비전공자가 AI 시대에 가져야 할 현실적인 학습 전략

## 6.1 문법을 완벽히 외우려고 하지 말자

AI 시대에는 문법을 모두 외우는 것보다 문법의 의미를 이해하는 것이 중요하다.

예를 들어 반복문의 정확한 형태가 기억나지 않을 수 있다. 그럴 때 AI나 검색을 사용할 수 있다.

하지만 다음은 알고 있어야 한다.

```text
반복문은 같은 작업을 여러 번 실행할 때 쓴다.
반복 조건을 잘못 쓰면 무한 반복이 생길 수 있다.
반복 중에 어떤 값이 변하는지 확인해야 한다.
```

외우는 양은 줄어들 수 있지만, 이해해야 하는 깊이는 사라지지 않는다.

## 6.2 코드를 직접 쳐보는 과정은 여전히 필요하다

AI가 코드를 만들어주더라도 초보자는 직접 쳐보고 실행해봐야 한다.

직접 쳐보면 다음을 경험한다.

- 괄호 하나가 빠져도 에러가 난다.
- 들여쓰기가 의미를 가진다.
- 파일 위치가 중요하다.
- 변수 이름이 맞아야 한다.
- 실행 순서가 중요하다.

이런 감각은 눈으로 보기만 해서는 잘 생기지 않는다.

처음에는 느리게 직접 작성하는 시간이 필요하다. 나중에 AI를 쓰더라도 이 경험이 있어야 결과를 판단할 수 있다.

## 6.3 작은 프로젝트를 완성해보자

비전공자에게 가장 중요한 경험은 작은 완성이다.

처음부터 거대한 서비스를 만들려고 하면 어렵다.

다음 정도의 작은 프로젝트가 좋다.

- 계산기
- 할 일 목록
- 회원 목록 관리
- 게시판
- 출석 체크
- 간단한 가계부
- 도서 대여 관리

중요한 것은 기능이 화려한 것이 아니라 처음부터 끝까지 연결해보는 것이다.

```text
화면을 만든다.
입력값을 받는다.
조건을 검사한다.
서버로 보낸다.
데이터베이스에 저장한다.
다시 조회해서 화면에 보여준다.
오류 상황을 처리한다.
```

이 흐름을 한 번이라도 완성해본 사람은 AI를 훨씬 실용적으로 사용할 수 있다.

## 6.4 AI에게 답을 맡기지 말고 설명을 요구하자

AI를 사용할 때는 “코드만 달라”고 하기보다 설명을 함께 요구하는 것이 좋다.

예:

```text
이 코드를 초보자도 이해할 수 있게 줄별로 설명해줘.
이 코드에서 에러가 날 수 있는 상황을 알려줘.
보안상 위험한 부분이 있는지 검토해줘.
더 작은 함수로 나눌 수 있는지 알려줘.
```

AI를 선생님처럼 활용할 수 있다. 하지만 최종 이해는 내 것이 되어야 한다.

## 6.5 기록하는 습관을 만들자

AI 시대에는 기록이 더 중요하다.

다음 내용을 짧게라도 정리해두면 좋다.

- 오늘 배운 개념
- 해결한 에러
- AI에게 물어본 질문
- AI 답변 중 맞았던 것
- AI 답변 중 틀렸던 것
- 다시 사용할 수 있는 코드 패턴

기록은 실력을 눈에 보이게 만든다. 취업 준비 과정에서도 프로젝트 설명과 포트폴리오 정리에 도움이 된다.

# 7장. AI 시대에 개발자 역할은 어떻게 바뀌는가?

## 7.1 코드 작성자에서 문제 해결자로

예전의 개발자 이미지는 “코드를 많이 치는 사람”이었다.

AI 시대의 개발자는 점점 “문제를 해결하는 사람”에 가까워진다.

```text
예전:
어떻게 코드를 작성할까?

현재와 앞으로:
무엇이 문제인가?
어떤 구조가 적절한가?
AI가 만든 코드가 맞는가?
어떻게 서비스로 완성할 것인가?
```

코드 작성 능력은 여전히 필요하다. 다만 코드 작성만으로는 부족하다.

## 7.2 혼자 다 외우는 사람보다 잘 연결하는 사람

현대 개발은 연결의 연속이다.

- 프론트엔드와 백엔드 연결
- 백엔드와 데이터베이스 연결
- 외부 API 연결
- 로그인 서비스 연결
- 결제 서비스 연결
- 배포 환경 연결

AI가 각 부분의 예제를 만들어도, 전체 연결 흐름을 이해해야 서비스가 완성된다.

앞으로는 “모든 문법을 외운 사람”보다 “필요한 지식을 찾아서 연결하고 검증할 수 있는 사람”이 더 중요해진다.

## 7.3 AI를 쓰는 개발자와 안 쓰는 개발자

AI를 무조건 거부할 필요는 없다. AI는 강력한 도구다.

다만 AI를 쓰는 방식이 중요하다.

좋은 활용:

- 반복 코드를 빠르게 작성한다.
- 에러 메시지를 해석한다.
- 테스트 케이스 아이디어를 얻는다.
- 낯선 개념의 첫 설명을 듣는다.
- 코드 리뷰 관점으로 검토를 요청한다.

위험한 활용:

- 이해하지 못한 코드를 그대로 붙여 넣는다.
- 보안 검토 없이 사용한다.
- 프로젝트 구조를 무시하고 적용한다.
- 실행 결과를 확인하지 않는다.
- AI 답변을 공식 문서보다 더 믿는다.

AI는 개발자를 대체하는 마법 상자가 아니라, 개발자의 능력을 확대하는 도구로 보는 것이 현실적이다.

## 7.4 초보자에게 더 중요한 태도

비전공자는 처음부터 모든 것을 잘할 수 없다. 중요한 것은 방향이다.

AI 시대에 초보자가 가져야 할 태도는 다음과 같다.

- 모르는 것을 부끄러워하지 않는다.
- 에러 메시지를 읽는다.
- AI 답변을 검토한다.
- 작은 기능부터 완성한다.
- 왜 이렇게 동작하는지 질문한다.
- 결과를 실행해서 확인한다.
- 기록하고 반복한다.

이 태도는 기술 변화가 있어도 오래 간다.

# 8장. 국비학원 수강생에게 필요한 현실적인 시선

## 8.1 취업이 쉬워졌다는 뜻은 아니다

AI가 있다고 해서 개발 공부가 쉬운 길이 되는 것은 아니다. 오히려 단순한 수준의 코드 작성만으로는 차별화가 어려워질 수 있다.

따라서 과장된 기대는 조심해야 한다.

```text
몇 개월 배우면 무조건 취업된다.
AI만 잘 쓰면 개발자가 된다.
비전공자도 금방 고연봉이 된다.
```

이런 표현은 현실적이지 않다.

하지만 반대로 “이제 시작해도 아무 의미 없다”는 말도 지나치다.

현실적인 관점은 다음과 같다.

```text
기초를 탄탄히 배우고,
작은 프로젝트를 완성하고,
AI를 도구로 활용하며,
자신이 만든 결과물을 설명할 수 있다면
개발자로 성장할 가능성은 여전히 있다.
```

## 8.2 수업에서 배워야 할 것은 문법만이 아니다

국비학원 과정에서 문법 수업은 중요하다. 하지만 문법 자체가 최종 목적은 아니다.

수업에서 함께 길러야 할 능력은 다음과 같다.

- 문제를 작은 단계로 나누기
- 코드 실행 흐름 이해하기
- 에러 메시지 읽기
- 데이터 구조 생각하기
- 화면과 서버 연결하기
- 데이터베이스 사용하기
- 프로젝트를 설명하기
- AI 답변 검토하기

이 능력들이 쌓이면 AI를 더 잘 사용할 수 있다.

## 8.3 포트폴리오도 달라져야 한다

AI 시대의 포트폴리오는 “코드를 많이 썼다”만으로는 부족하다.

다음이 드러나야 한다.

- 어떤 문제를 해결하려 했는가?
- 어떤 기능을 만들었는가?
- 데이터베이스 구조는 어떻게 설계했는가?
- 어려웠던 오류는 무엇이고 어떻게 해결했는가?
- 보안이나 예외 처리는 어떻게 고려했는가?
- AI를 사용했다면 어떤 부분에 사용했고 어떻게 검토했는가?
- 본인이 직접 이해하고 설명할 수 있는 부분은 무엇인가?

AI를 사용했다는 사실 자체가 문제가 되는 것은 아니다. 중요한 것은 이해와 검증이다.

## 8.4 면접에서 중요한 질문

AI 시대의 면접에서는 단순히 “이 코드를 작성했나요?”보다 다음 질문이 더 중요해질 수 있다.

- 이 기능의 흐름을 설명해보세요.
- 왜 이런 테이블 구조를 선택했나요?
- 이 API에서 예외 상황은 무엇인가요?
- 로그인 기능에서 보안상 조심할 점은 무엇인가요?
- 이 오류를 어떻게 찾고 해결했나요?
- AI를 사용했다면 어떤 식으로 검토했나요?
- 이 코드를 다시 만든다면 무엇을 개선하고 싶나요?

이 질문에 답하려면 프로젝트를 직접 이해해야 한다.

# 9장. AI와 함께 공부하는 실습 방법

## 9.1 실습 1: AI 코드 읽기

AI에게 다음 요청을 해보자.

```text
Python으로 점수가 60점 이상이면 합격, 아니면 불합격을 출력하는 코드를 작성해줘.
```

그다음 다음 질문에 답해보자.

1. 변수는 무엇인가?
2. 조건문은 어디인가?
3. 조건이 참일 때 실행되는 코드는 무엇인가?
4. 조건이 거짓일 때 실행되는 코드는 무엇인가?
5. 점수가 60점일 때 결과는 무엇인가?

목표는 코드를 받는 것이 아니라 코드를 읽는 것이다.

## 9.2 실습 2: AI 코드 수정하기

AI가 만든 코드에서 일부 조건을 바꿔보자.

```text
90점 이상은 A
80점 이상은 B
70점 이상은 C
60점 이상은 D
그 외는 F
```

수정한 뒤 다음을 확인한다.

- 95점은 A가 나오는가?
- 80점은 B가 나오는가?
- 59점은 F가 나오는가?
- 조건의 순서를 바꾸면 결과가 달라지는가?

AI가 만든 코드를 수정할 수 있어야 내 코드가 된다.

## 9.3 실습 3: 요구사항을 구체화하기

처음 요구사항:

```text
할 일 목록 앱을 만들어줘.
```

이 요구사항을 더 구체적으로 바꿔보자.

```text
사용자는 할 일을 추가할 수 있다.
사용자는 할 일을 완료 처리할 수 있다.
사용자는 완료된 할 일을 삭제할 수 있다.
할 일에는 제목과 작성일이 있다.
제목은 비어 있으면 안 된다.
완료된 할 일은 목록에서 취소선으로 표시한다.
```

이렇게 구체화한 뒤 AI에게 요청하면 더 좋은 결과를 얻을 수 있다.

## 9.4 실습 4: AI 답변 검토하기

AI에게 회원가입 기능 코드를 요청한 뒤 다음 항목을 검토해보자.

- 이메일 중복 검사를 하는가?
- 비밀번호를 그대로 저장하지 않는가?
- 비밀번호 길이를 검사하는가?
- 실패했을 때 적절한 메시지를 주는가?
- 데이터베이스 저장 과정이 있는가?
- 예외 상황을 처리하는가?

초보자에게 어려울 수 있지만, 이런 관점으로 코드를 보는 습관이 중요하다.

## 9.5 실습 5: 에러 질문 잘하기

다음 나쁜 질문을 좋은 질문으로 바꿔보자.

나쁜 질문:

```text
코드가 안 돼요.
```

좋은 질문에 포함할 내용:

- 사용한 언어와 프레임워크
- 어떤 기능을 만들고 있었는지
- 기대한 동작
- 실제 동작
- 에러 메시지
- 관련 코드 일부
- 이미 시도해본 해결 방법

예시:

```text
Java Spring Boot로 게시글 작성 API를 만들고 있습니다.
POST /posts 요청을 보내면 게시글이 저장되길 기대했습니다.
하지만 500 에러가 발생하고, 콘솔에는 Table 'board.posts' doesn't exist 라고 나옵니다.
현재 MySQL을 사용하고 있고, posts 테이블을 아직 만들지 않았을 수 있습니다.
어떤 순서로 확인하면 좋을까요?
```

이런 질문은 AI뿐 아니라 사람에게 물어볼 때도 도움이 된다.

# 11장. 핵심 정리

## 11.1 개발자는 사라진 것이 아니라 역할이 바뀌고 있다

AI는 코드를 만들 수 있다. 하지만 개발은 코드 생성만으로 끝나지 않는다.

문제 정의, 요구사항 분석, 구조 설계, 코드 검토, 오류 수정, 보안 판단, 데이터베이스와 API 연결, 프론트엔드와 백엔드 통합, 배포와 운영은 여전히 사람이 이해해야 한다.

## 11.2 프로그래밍을 배우는 이유도 바뀌고 있다

예전에는 코드를 직접 작성하기 위해 프로그래밍을 배웠다.

지금은 여기에 더해 AI가 만든 코드를 읽고, 검토하고, 수정하고, 실제 서비스에 맞게 연결하기 위해 프로그래밍을 배운다.

## 11.3 AI는 초보자를 대신해주지 않는다

AI는 초보자의 학습을 도와줄 수 있다. 하지만 이해하지 못한 코드를 대신 책임져주지는 않는다.

AI를 잘 쓰는 사람은 기본기가 있는 사람이다.

## 11.4 비전공자에게 필요한 것은 완벽함이 아니라 방향이다

비전공자가 처음부터 모든 것을 알 필요는 없다.

하지만 다음 방향은 분명히 잡아야 한다.

- 기본 문법을 이해한다.
- 코드를 직접 실행해본다.
- 에러 메시지를 읽는다.
- 작은 프로젝트를 완성한다.
- 데이터베이스와 API 흐름을 이해한다.
- AI 답변을 검토하는 습관을 가진다.
- 자신이 만든 것을 설명할 수 있게 만든다.

## 11.5 현실적인 결론

AI 때문에 개발 공부가 의미 없어지는 것은 아니다.

다만 예전처럼 단순히 문법만 외우고 예제만 따라 하는 방식으로는 부족하다.

앞으로의 개발자는 AI를 쓰지 않는 사람이 아니라, AI를 이해하고 검토하며 실제 문제 해결에 연결할 수 있는 사람이어야 한다.

프로그래밍을 배운다는 것은 AI와 경쟁하기 위한 것이 아니라, AI를 제대로 사용할 수 있는 사람이 되기 위한 과정이다.

# 부록. AI 코드 검토 체크리스트

AI가 작성한 코드를 사용할 때는 다음 항목을 확인하자.

| 체크 항목 | 확인 질문 |
|---|---|
| 실행 여부 | 실제로 실행되는가? |
| 요구사항 일치 | 내가 원한 기능과 맞는가? |

| 예외 처리 | 오류 상황에서 안전하게 동작하는가? |

| 데이터 흐름 | 화면, 서버, DB 연결이 맞는가? |

| 설명 가능성 | 내가 이 코드를 설명할 수 있는가? |

마지막 항목이 특히 중요하다.

```text
설명할 수 없는 코드는 내 코드가 아니다.
```

# 마무리

AI 시대에 개발을 시작하는 것은 불안할 수 있다. 그 불안은 가볍게 넘길 문제가 아니다. 실제로 개발자의 일은 바뀌고 있고, 단순 반복 코딩의 가치는 줄어들고 있다.

하지만 개발의 본질은 여전히 남아 있다.

누군가는 문제를 정의해야 한다. 누군가는 요구사항을 정리해야 한다. 누군가는 데이터와 화면과 서버를 연결해야 한다. 누군가는 AI가 만든 코드가 맞는지 확인해야 한다. 누군가는 오류를 고치고, 보안을 점검하고, 실제 사용 가능한 서비스로 완성해야 한다.

그 일을 하기 위해 프로그래밍을 배운다.

AI는 개발 공부의 끝이 아니라, 개발 공부의 이유를 바꾸고 있다.
